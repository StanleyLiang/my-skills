# React Router v7 (SPA mode) reference

## Library vs framework mode

This skill uses **library mode** (also called SPA mode): you import from `react-router-dom`, build your own bundler config (rsbuild here), and mount `<RouterProvider>` yourself. Framework mode (Remix-style with `react-router` CLI) is NOT used.

## Entrypoint shape

```tsx
// src/main.tsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import { createBrowserRouter, RouterProvider } from 'react-router-dom';
import './index.css';
import { routes } from './routes.gen';

const router = createBrowserRouter(routes);

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <RouterProvider router={router} />
  </StrictMode>,
);
```

`routes.gen.ts` is generated by `wire-entrypoint.mjs` from the `route.meta.json` files emitted during route porting.

## Route configuration shape

```ts
// routes.gen.ts (excerpt)
import { RouteObject } from 'react-router-dom';

export const routes: RouteObject[] = [
  {
    path: '/',
    Component: lazy(() => import('./routes/_layout')),
    HydrateFallback: () => null,
    errorElement: <Error />,
    children: [
      { index: true, Component: lazy(() => import('./routes/page')) },
      { path: 'foo/:id', Component: lazy(() => import('./routes/foo/page')) },
      { path: '*', Component: lazy(() => import('./routes/not-found')) },
    ],
  },
];
```

## Hook equivalents (Next/RR)

| Behavior | RR hook |
|---|---|
| current path | `useLocation()` → `.pathname` |
| query string | `useSearchParams()` (returns `[params, set]`) |
| dynamic params | `useParams<{...}>()` |
| programmatic nav | `useNavigate()` |
| matched route data | `useLoaderData()` |
| revalidate | `useRevalidator()` |
| navigation state | `useNavigation()` (`.state` is `'idle' | 'loading' | 'submitting'`) |

## `<Link>`

```tsx
import { Link, NavLink } from 'react-router-dom';

<Link to="/foo">Foo</Link>
<Link to={{ pathname: '/foo', search: '?x=1' }}>Foo</Link>
<NavLink to="/foo" className={({ isActive }) => isActive ? 'active' : ''} />
```

## Loaders & actions (data router)

Optional but recommended. Per-route:

```ts
{
  path: 'users/:id',
  loader: async ({ params }) => {
    const r = await fetch(`/api/users/${params.id}`);
    if (!r.ok) throw new Response(null, { status: r.status });
    return r.json();
  },
  Component: User,
  errorElement: <UserError />,
}
```

`port-routes.mjs` does NOT auto-create loaders — loaders need backend knowledge the script doesn't have. Anything that was a server-side `fetch` in an RSC stays in the component as `useEffect` + `useState` with a TODO to consider moving to a loader.

## v6 → v7 deltas (if user picks v6 instead)

- `json` helper removed in v7 — use `Response.json()` or `new Response(JSON.stringify(x), {...})`.
- Future flags from v6 (`v7_normalizeFormMethod`, `v7_fetcherPersist`, `v7_relativeSplatPath`, etc.) are default in v7 — drop the `future` opt-in object.
- `useNavigate()` returns a stable function in v7 (it was unstable in some v6 minors).
- `Form` from `react-router-dom` is the only `Form` (no longer separate from `react-router-dom/server`).

## Scroll restoration

```tsx
import { ScrollRestoration } from 'react-router-dom';

// inside the root layout
<>
  <Outlet />
  <ScrollRestoration />
</>
```

Place once at the root layout.

## Catch-all and 404

```ts
{ path: '*', Component: NotFound }
```

at the top of the routes list (or as the last child of the root) handles unmatched paths.

## Lazy loading routes

```ts
import { lazy } from 'react';
const Page = lazy(() => import('./routes/foo/page'));
```

`createBrowserRouter` accepts the lazy component via the `lazy` field of a route object too:

```ts
{ path: '/foo', lazy: () => import('./routes/foo/page').then(m => ({ Component: m.default })) }
```

The skill prefers `Component: lazy(...)` because it's friendlier to TypeScript.
